# Sequence Type Transitions (serial ↔ GENERATED AS IDENTITY)

> pgschema issue [#279](https://github.com/pgplex/pgschema/issues/279) (closed)

## Context

PostgreSQL supports several auto-increment styles with materially different
catalog state and DDL:

- legacy `serial` / `bigserial`
- `GENERATED ALWAYS AS IDENTITY`
- `GENERATED BY DEFAULT AS IDENTITY`

Transitions between them are not cosmetic. They require the planner to handle
sequence ownership, column defaults, identity metadata, and operation ordering
correctly. pgschema issue #279 was about missing or incorrect handling of those
transitions.

This parity gap is now fixed in pg-delta. The linked pg-toolbelt work landed in
[pg-toolbelt#154](https://github.com/supabase/pg-toolbelt/pull/154), and the
current test suite covers both creation of identity columns and transitions
between serial and identity styles.

## Reproduction SQL

```sql
CREATE SCHEMA test_schema;

CREATE TABLE test_schema.items (
    c1 int NOT NULL,
    c2 serial,
    c3 int GENERATED ALWAYS AS IDENTITY
);
```

**Change to diff:**

```sql
CREATE SEQUENCE test_schema.items_c1_seq OWNED BY test_schema.items.c1;
ALTER TABLE test_schema.items ALTER COLUMN c1 SET DEFAULT nextval('test_schema.items_c1_seq'::regclass);
ALTER TABLE test_schema.items ALTER COLUMN c2 DROP DEFAULT;
DROP SEQUENCE test_schema.items_c2_seq;
ALTER TABLE test_schema.items ALTER COLUMN c2 ADD GENERATED ALWAYS AS IDENTITY;
ALTER TABLE test_schema.items ALTER COLUMN c3 SET GENERATED BY DEFAULT;
```

## How pgschema handled it

pgschema fixed the issue by tracking identity/sequence state explicitly enough
to emit the proper transition DDL instead of treating serial and identity as the
same column shape.

## Current pg-delta status

| Aspect | Status |
|---|---|
| `GENERATED ALWAYS AS IDENTITY` creation | ✅ Covered in `tests/integration/sequence-operations.test.ts` |
| `GENERATED BY DEFAULT AS IDENTITY` creation | ✅ Covered in `tests/integration/sequence-operations.test.ts` |
| serial → identity transition | ✅ Covered by `"serial and identity transition diffs"` |
| identity → serial transition | ✅ Covered by `"identity to serial transition diffs"` |
| mode change `ALWAYS` ↔ `BY DEFAULT` | ✅ Covered by the same transition suite |
| Existing pg-toolbelt issue / PR | ✅ [#138](https://github.com/supabase/pg-toolbelt/issues/138) closed by [#154](https://github.com/supabase/pg-toolbelt/pull/154) |

Current integration evidence includes:

- `"create table with GENERATED ALWAYS AS IDENTITY column"`
- `"create table with GENERATED BY DEFAULT AS IDENTITY column"`
- `"serial and identity transition diffs"`
- `"identity to serial transition diffs"`

Those tests assert the expected operations such as:

- `ALTER TABLE ... DROP DEFAULT`
- `DROP SEQUENCE ... CASCADE`
- `ALTER TABLE ... ADD GENERATED ALWAYS AS IDENTITY`
- `ALTER TABLE ... SET GENERATED BY DEFAULT`
- `ALTER TABLE ... DROP IDENTITY`

## Comparison of approaches

| | pgschema | pg-delta |
|---|---|---|
| **Historical gap** | Identity transitions were not fully tracked | Same class of gap existed |
| **Current handling** | Fixed in upstream issue/PR flow | Fixed with explicit diff + ordering coverage |
| **Regression evidence** | pgschema fixtures | pg-delta integration tests assert transition SQL terms |

## Resolution in pg-delta

pg-delta now handles serial/identity transitions as a solved parity case rather
than a missing edge case. The benchmark entry is retained for historical
context, but no active gap remains for this scenario in the refreshed
`c7e97f8` snapshot.
