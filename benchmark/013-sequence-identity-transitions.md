# Sequence Type Transitions (serial <-> GENERATED AS IDENTITY)

> pgschema issue [#279](https://github.com/pgplex/pgschema/issues/279) (closed), fixed by [pgschema#282](https://github.com/pgplex/pgschema/pull/282)

## Context

PostgreSQL supports multiple auto-increment styles:

- `serial` / `bigserial`
- `GENERATED ALWAYS AS IDENTITY`
- `GENERATED BY DEFAULT AS IDENTITY`

Converting between these styles is not a cosmetic rewrite. It changes how
sequences are owned, how defaults are attached, and whether callers can
provide explicit values.

pgschema issue #279 covered missing or incorrect handling for these
transitions. pg-delta used to lag here as well, but the parity gap is now
fixed by [pg-toolbelt#154](https://github.com/supabase/pg-toolbelt/pull/154),
which closed [pg-toolbelt#138](https://github.com/supabase/pg-toolbelt/issues/138).

## Reproduction SQL

```sql
CREATE SCHEMA test_schema;

CREATE TABLE test_schema.items (
    c1 int NOT NULL,
    c2 serial,
    c3 int GENERATED ALWAYS AS IDENTITY
);
```

**Change to diff:**

```sql
CREATE SEQUENCE test_schema.items_c1_seq OWNED BY test_schema.items.c1;
ALTER TABLE test_schema.items ALTER COLUMN c1 SET DEFAULT nextval('test_schema.items_c1_seq'::regclass);
ALTER TABLE test_schema.items ALTER COLUMN c2 DROP DEFAULT;
DROP SEQUENCE test_schema.items_c2_seq;
ALTER TABLE test_schema.items ALTER COLUMN c2 ADD GENERATED ALWAYS AS IDENTITY;
ALTER TABLE test_schema.items ALTER COLUMN c3 SET GENERATED BY DEFAULT;
```

## How pgschema handled it

pgschema improved its identity / sequence tracking so it can detect the
current style and emit the correct transition DDL for serial columns,
identity columns, and `ALWAYS` vs `BY DEFAULT` changes.

## Current pg-delta status

| Aspect | Status |
|---|---|
| Serial column sequence dependency coverage | Yes |
| `GENERATED ALWAYS AS IDENTITY` coverage | Yes - `create table with GENERATED ALWAYS AS IDENTITY column` |
| `GENERATED BY DEFAULT AS IDENTITY` coverage | Yes - `create table with GENERATED BY DEFAULT AS IDENTITY column` |
| Serial / identity transition coverage | Yes - `serial and identity transition diffs` |
| Identity column model fields | Yes - `src/core/objects/table/table.model.ts` extracts `is_identity` and `is_identity_always` from `pg_attribute.attidentity` |
| Existing pg-toolbelt issue / PR | Yes - issue [#138](https://github.com/supabase/pg-toolbelt/issues/138) closed by merged PR [#154](https://github.com/supabase/pg-toolbelt/pull/154) |

The integration coverage lives in
`repos/pg-toolbelt/packages/pg-delta/tests/integration/sequence-operations.test.ts`.
The transition test asserts the expected `DROP DEFAULT`, `DROP SEQUENCE`,
`ADD GENERATED ALWAYS AS IDENTITY`, and `SET GENERATED BY DEFAULT`
operations for the mixed serial / identity scenario.

## Comparison of approaches

| | pgschema | pg-delta |
|---|---|---|
| Historical root cause | Incomplete identity / sequence tracking | Same class of gap |
| Current upstream state | Fixed in merged PR #282 | Fixed in merged PR #154 |
| Regression coverage | pgschema diff fixtures | Dedicated roundtrip tests for identity creation and transition diffs |

## Resolution in pg-delta

pg-delta now covers the benchmarked serial / identity transitions and the
supporting identity metadata is present in the table model and diff logic.
This benchmark item is solved in the current snapshot.
