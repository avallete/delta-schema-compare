# Sequence Type Transitions (serial ↔ GENERATED AS IDENTITY)

> pgschema issue [#279](https://github.com/pgplex/pgschema/issues/279) (closed)

## Context

PostgreSQL supports several auto-increment styles:

- `serial` / `bigserial`
- `GENERATED ALWAYS AS IDENTITY`
- `GENERATED BY DEFAULT AS IDENTITY`

Moving between these styles requires different DDL. For example, `serial` →
identity needs the old default removed before identity is added, and identity →
`serial` needs the implicit identity sequence torn down before a new owned
sequence/default pair is created.

pgschema originally missed some of these transitions, which is what issue #279
fixed upstream.

Refresh note (2026-05-07): this parity gap is now fixed in pg-delta. The
tracking issue [pg-toolbelt#138](https://github.com/supabase/pg-toolbelt/issues/138)
is closed, and
[pg-toolbelt#154](https://github.com/supabase/pg-toolbelt/pull/154) merged the
identity-transition logic plus focused integration coverage.

## Reproduction SQL

```sql
CREATE SCHEMA test_schema;

CREATE TABLE test_schema.items (
  c1 int NOT NULL,
  c2 serial,
  c3 int GENERATED ALWAYS AS IDENTITY
);
```

**Change to diff:**

```sql
CREATE TABLE test_schema.items (
  c1 serial NOT NULL,
  c2 int GENERATED ALWAYS AS IDENTITY,
  c3 int GENERATED BY DEFAULT AS IDENTITY
);
```

## How pgschema handled it

pgschema improved its sequence/identity tracking so it can distinguish the
different auto-increment styles and emit the correct transition DDL.
