# Sequence Type Transitions (serial ↔ GENERATED AS IDENTITY)

> pgschema issue [#279](https://github.com/pgplex/pgschema/issues/279) (closed)

## Context

PostgreSQL exposes several auto-increment styles:
- `serial` / `bigserial` (legacy sequence-backed defaults)
- `GENERATED ALWAYS AS IDENTITY`
- `GENERATED BY DEFAULT AS IDENTITY`

Transitions between them are not cosmetic. PostgreSQL uses different DDL for
`ADD GENERATED ... AS IDENTITY`, `SET GENERATED ...`, and `DROP IDENTITY`, and
the planner also has to preserve or clear the old serial-style default at the
right time.

pgschema originally failed to track these transitions correctly. This benchmark
entry is now historical for pg-delta: the current implementation extracts
identity metadata from `pg_attribute.attidentity`, diffs it, and covers the
relevant transitions with dedicated integration tests. The original pg-toolbelt
tracking issue [#138](https://github.com/supabase/pg-toolbelt/issues/138) is
closed, and [pg-toolbelt#154](https://github.com/supabase/pg-toolbelt/pull/154)
merged the parity work.

## Reproduction SQL

```sql
CREATE SCHEMA test_schema;

CREATE TABLE test_schema.items (
    c1 int NOT NULL,
    c2 serial,
    c3 int GENERATED ALWAYS AS IDENTITY
);
```

**Change to diff:**

```sql
CREATE TABLE test_schema.items (
    c1 serial NOT NULL,
    c2 int GENERATED ALWAYS AS IDENTITY,
    c3 int GENERATED BY DEFAULT AS IDENTITY
);
```

**Expected DDL:**

```sql
-- c1: add serial (create sequence + set default)
CREATE SEQUENCE test_schema.items_c1_seq OWNED BY test_schema.items.c1;
ALTER TABLE test_schema.items ALTER COLUMN c1 SET DEFAULT nextval('test_schema.items_c1_seq');

-- c2: serial → IDENTITY (drop default, drop sequence, add identity)
ALTER TABLE test_schema.items ALTER COLUMN c2 DROP DEFAULT;
DROP SEQUENCE test_schema.items_c2_seq;
ALTER TABLE test_schema.items ALTER COLUMN c2 ADD GENERATED ALWAYS AS IDENTITY;

-- c3: ALWAYS → BY DEFAULT
ALTER TABLE test_schema.items ALTER COLUMN c3 SET GENERATED BY DEFAULT;
```

## How pgschema handled it

pgschema improved its sequence tracking to detect all three identity styles and
generate the correct transition DDL.

## Current pg-delta status

| Aspect | Status |
|---|---|
| Identity metadata extracted from catalog | ✅ `src/core/objects/table/table.model.ts` reads `a.attidentity` into `is_identity` / `is_identity_always` |
| Diff detects serial/identity transitions | ✅ `src/core/objects/table/table.diff.ts` compares identity flags and emits the appropriate changes |
| `ADD GENERATED ... AS IDENTITY` serializer | ✅ `src/core/objects/table/changes/table.alter.ts` |
| `SET GENERATED ...` serializer | ✅ Same file |
| `DROP IDENTITY` serializer | ✅ Same file |
| `GENERATED ALWAYS AS IDENTITY` create coverage | ✅ `tests/integration/sequence-operations.test.ts` |
| `GENERATED BY DEFAULT AS IDENTITY` create coverage | ✅ Same file |
| serial -> identity / ALWAYS -> BY DEFAULT coverage | ✅ Same file |
| identity -> serial coverage | ✅ Same file (`DROP IDENTITY`) |
| Existing pg-toolbelt issue / PR | ✅ Issue [#138](https://github.com/supabase/pg-toolbelt/issues/138) closed by merged PR [#154](https://github.com/supabase/pg-toolbelt/pull/154) |

## Comparison of approaches

| | pgschema | pg-delta |
|---|---|---|
| **Historical gap** | Incomplete transition tracking | Same class of gap |
| **Current modeling** | Fixed upstream | Identity flags extracted directly from catalog metadata |
| **Current coverage** | Resolved upstream | Dedicated integration cases for create and conversion flows |

## Resolution in pg-delta

This benchmark entry is retained as historical context, but the parity gap is
now solved in the current pg-delta snapshot.
