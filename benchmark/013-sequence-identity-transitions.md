# Sequence Type Transitions (serial ↔ GENERATED AS IDENTITY)

> pgschema issue [#279](https://github.com/pgplex/pgschema/issues/279) (closed)

## Context

PostgreSQL supports three distinct auto-increment styles:

- `serial` / `bigserial`
- `GENERATED ALWAYS AS IDENTITY`
- `GENERATED BY DEFAULT AS IDENTITY`

Transitions between those styles are not cosmetic. PostgreSQL needs different
DDL for `serial -> identity`, `identity -> serial`, and `ALWAYS <-> BY DEFAULT`
changes.

pgschema originally missed some of those transitions.

Refresh note (2026-05-13): this parity gap is now fixed in pg-delta. The
tracking issue [pg-toolbelt#138](https://github.com/supabase/pg-toolbelt/issues/138)
is closed, and
[pg-toolbelt#154](https://github.com/supabase/pg-toolbelt/pull/154) added both
the diff logic and the roundtrip coverage for identity-aware transitions.

## Reproduction SQL

```sql
CREATE SCHEMA test_schema;

CREATE TABLE test_schema.items (
    c1 int NOT NULL,
    c2 serial,
    c3 int GENERATED ALWAYS AS IDENTITY
);
```

**Change to diff:**

```sql
CREATE TABLE test_schema.items (
    c1 serial NOT NULL,
    c2 int GENERATED ALWAYS AS IDENTITY,
    c3 int GENERATED BY DEFAULT AS IDENTITY
);
```

## How pgschema handled it

pgschema improved its column / sequence tracking so it can distinguish legacy
serial defaults from SQL-standard identity columns and emit the correct
transition DDL for each direction.

| Aspect | Status |

| `GENERATED BY DEFAULT AS IDENTITY` create coverage | ✅ Present |
| `serial -> identity` transition coverage | ✅ Present |
| `identity -> serial` transition coverage | ✅ Present |
| `ALWAYS -> BY DEFAULT` transition coverage | ✅ Present |
| Existing pg-toolbelt issue / PR | ✅ [#138](https://github.com/supabase/pg-toolbelt/issues/138) closed by merged PR [#154](https://github.com/supabase/pg-toolbelt/pull/154) |

Current integration coverage now exercises:

- `"create table with GENERATED ALWAYS AS IDENTITY column"`
- `"create table with GENERATED BY DEFAULT AS IDENTITY column"`
- `"serial and identity transition diffs"`
- `"identity to serial transition diffs"`

Those cases verify the exact statement families that used to be missing,
including `ADD GENERATED ALWAYS AS IDENTITY`, `SET GENERATED BY DEFAULT`,
`DROP IDENTITY`, and synthetic sequence recreation for the `identity -> serial`
path.

## Comparison of approaches

| | pgschema | pg-delta |
|---|---|---|
| **Historical root cause** | Incomplete handling of identity-style transitions | Same class of gap in diff output and coverage |
| **Current fix** | Distinguish serial vs identity styles in IR/diff planning | Emit explicit identity add/set/drop and sequence recreation flows |
| **Regression coverage** | Upstream fixtures for transition cases | Focused integration coverage for create and both transition directions |

## Resolution in pg-delta

No active parity gap remains for this benchmark item in the current pg-delta
snapshot. The benchmark entry now mainly documents why identity transitions are
handled as first-class schema changes rather than treated like ordinary default
updates.
